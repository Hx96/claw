## Cron 推送投递问题（第3次复发）

### 问题
- 用户08:08反馈"今日任务没收到"
- 排查发现：4个任务正常执行（status=ok），但 `deliveryStatus=not-delivered`
- 根因：`delivery.mode` 被重置为 `none`（jobs.json 今天08:01被重建，可能是gateway写入）
- 这是第3次出现 delivery.mode 重置问题（04-11、07-20、07-21）

### 修复措施
1. 5个任务 `delivery.mode` 从 `none` 改回 `announce`
2. **关键改进**：删除所有 cron prompt 中的"手动调 message 工具投递"指令
   - 之前 prompt 里有 `## 投递验证` 段落，让 AI 在隔离会话中手动调 message
   - 这与 `delivery.mode=announce` 冲突，且隔离会话调 message 不可靠
   - 现在统一让系统 `announce` 投递，AI 只生成内容
3. 手动补发今日内容（AI HOT + 美股日报 + AI Builders 简报）

### 待观察
- delivery.mode 重置为 none 的根因尚未找到。jobs.json 在 08:01 被重建（可能是 gateway 写入时丢失 delivery 配置）
- 需要持续观察明天是否再次重置
- 如果再次重置，需要排查 gateway 的 cron job persistence 逻辑

### 教训更新
- cron prompt 不应包含手动调 message 的指令 → 系统层 announce 投递是唯一可靠路径
- 隔离会话调 message 工具虽然能拿到 messageId，但消息可能被微信折叠/吞掉
- delivery.mode 会在某些条件下被重置（可能是 gateway 重启、jobs.json 重建），需要定期检查

## Cron 全面加固（09:58 用户要求优化）

通过 `openclaw cron edit` 正式配置所有7个任务：
1. `delivery.mode=announce` — 系统层投递，不依赖AI手动调message
2. `delivery.bestEffort=false` — 投递失败=任务失败（触发重试+告警）
3. `failureAlert.after=1` — 连续1次失败即发告警到微信
4. `failureAlert.channel=openclaw-weixin` — 告警发到当前微信
5. 所有prompt清除手动message指令
